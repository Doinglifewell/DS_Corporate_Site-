Platform Roles & Permissions

Roles & Permissions Architecture

Two systems that intersect. Platform roles control what users can do. DPP access tiers control what product data the public can see. Both are required for ESPR compliance.

1
Platform Roles
Who can do what inside DeStore.
Webmaster
Platform-level · DeStore internal
Full platform control. Manages all brands, configures system settings, overrides per-brand permissions, manages QR templates, edits result screen permissions, sets physical tag pricing. This is Josiah / DeStore admin team.
All brand access System config QR template library Result screen permissions Tag pricing User management Audit logs EU Registry admin
Brand Owner
Brand-level · Primary account holder
The brand's primary account. Creates collections and products, runs AI onboarding, configures DPP workflow, manages team members, invites compliance officers, views analytics, issues DPPs. Full control of their brand's data.
Create collections Create products AI onboarding DPP workflow Issue DPPs Invite team Invite compliance officer View reports Analytics Billing Branding settings
Brand Team Member
Brand-level · Invited by Brand Owner
Staff members at the brand. Can create products, run AI onboarding, generate reports, and manage day-to-day DPP operations. Cannot invite compliance officers, manage billing, or change brand settings.
Create products AI onboarding DPP workflow Issue DPPs View reports Invite compliance officer Billing Branding settings Invite team
Compliance Officer
Brand-level · External · Invited by Brand Owner
External reviewer invited by the brand. Can view DPP Preliminary Reports, review compliance data, approve or reject DPPs, and add reviewer notes. Cannot edit product data, create products, or access billing. Their approval is captured with timestamp, signature, and (future) KILT DID.
View DPP reports Review compliance data Approve / reject DPPs Add reviewer notes Digital signature KILT DID sign (future) Deloitte KYC (future) Edit product data Create products Billing Branding
Consumer
Public · No login required for Tier 1
End users who scan QR codes and interact with product DPPs. Tier 1 data (product info, materials, sustainability) is free and ungated. Logged-in consumers can claim ownership, trade, gift, recycle, and view their DeAR wallet.
View DPP (Tier 1) Scan QR codes Claim ownership Trade / Gift / Recycle DeAR wallet Edit product data View Tier 2/3 data Admin access
Trade Partner / Recycler
ESPR Tier 2 · Business auth required
Resellers, repair shops, and recyclers who need access to extended product data for circular economy operations. Sees Tier 1 + REACH details, Bill of Materials, supplier names, disassembly instructions. Requires business authentication.
View DPP (Tier 1 + 2) REACH / BoM data Supplier names Disassembly instructions Business verification Tier 3 data Compliance docs

DPP Access Tiers (ESPR RBAC)
What product data each audience can see when scanning a QR code. Required by EU ESPR Article 10.
Tier 1 · Public
No login required · Free and ungated
  • Product name, brand, images
  • Material composition summary
  • Recycled content percentage
  • Carbon footprint (kg CO₂e)
  • Country of manufacture
  • Durability rating
  • Repairability score
  • Recycling instructions
  • Circular actions (trade/gift/recycle)
  • Verified badge status
Tier 2 · Trade / Recycler
Business authentication required
  • Everything in Tier 1, plus:
  • Full material composition table
  • REACH / SVHC substance details
  • Bill of Materials reference
  • Supplier names (if disclosed)
  • Disassembly instructions (full)
  • Spare parts availability
  • Packaging composition
Tier 3 · Regulator
eIDAS authentication · Future (KILT DID)
  • Everything in Tier 1 + 2, plus:
  • Declaration of Conformity (PDF)
  • Full supply chain provenance
  • On-chain token IDs + tx hashes
  • Compliance officer approval record
  • W3C Verifiable Credential (future)
  • KILT DID verification (future)
  • EU DPP Registry submission status
4
Compliance Officer Invite Flow
How a brand invites an external reviewer and how the approval gets captured.
1
Brand Owner invites officer
Enters name, email, title, company. DeStore sends an invitation email with a unique link.
POST /api/invites { role: "compliance_officer", email, brandId }
2
Officer clicks email link
Lands on DeStore. Creates account (email + Google SSO) or signs in if they already have one. Account linked to brand with "compliance_officer" role.
Thirdweb inAppWallet → Smart Account created on first login
3
Officer reviews DPP report
Sees the DPP Preliminary Report with compliance score, traffic-light status per field, and all product data. Can add reviewer notes to specific sections.
GET /api/reports/{reportId} → scoped to compliance_officer role
4
Officer approves (or requests changes)
Clicks "Approve" → enters digital signature (drawn or typed). Or clicks "Request Changes" with notes on what needs fixing.
POST /api/reports/{reportId}/approve { signature, notes, timestamp }
5
Approval captured (3 layers)
Layer 1: Name, title, timestamp, SHA-256 report hash stored in DB. Layer 2: Event logged to Soneium SBT. Layer 3 (future): KILT DID + Deloitte KYC credential signs the approval.
n8n webhook → Soneium SBT event → (future) KILT DaaS API
6
Both parties notified
Brand owner receives email: "Your DPP has been approved by [officer name]." Officer receives confirmation email with receipt. DPP status changes to "Externally Verified ◈".
n8n email workflow → brand + officer

5
Data Model
Database schema for roles, invites, and approvals.
users
{
  id: uuid,
  email: string,
  name: string,
  walletAddress: string,    // Thirdweb smart account
  kiltDid: string | null,   // Future: KILT DID
  kiltKycVerified: boolean, // Future: Deloitte KYC
  createdAt: timestamp
}
brand_roles
{
  id: uuid,
  userId: uuid,
  brandId: uuid,
  role: enum [
    "owner",
    "team_member",
    "compliance_officer"
  ],
  title: string | null,     // Officer's title
  company: string | null,   // Officer's company
  invitedBy: uuid,
  invitedAt: timestamp,
  acceptedAt: timestamp | null
}
dpp_approvals
{
  id: uuid,
  reportId: uuid,
  officerId: uuid,          // → users.id
  brandRoleId: uuid,        // → brand_roles.id
  status: enum [
    "pending",
    "approved",
    "changes_requested",
    "rejected"
  ],
  signature: text,          // Digital signature data
  notes: text | null,
  reportHash: string,       // SHA-256 of report at approval
  onChainTxHash: string,    // Soneium event tx
  kiltCredentialId: string, // Future: KILT VC ID
  approvedAt: timestamp
}
invites
{
  id: uuid,
  brandId: uuid,
  email: string,
  role: enum,
  name: string,
  title: string | null,
  company: string | null,
  token: string,            // Unique invite token
  invitedBy: uuid,
  status: enum [
    "pending",
    "accepted",
    "expired",
    "revoked"
  ],
  expiresAt: timestamp,     // 30 days
  createdAt: timestamp
}
